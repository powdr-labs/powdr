# Problem:

The following things seem strange to me:
- When running in software (metrics_v2_pairing_apc000), the metrics viewer shows that there are 5.77B trace cells, but the APC analyzer reports the software cost (in terms of trace cells) as 1.5B.
- Similarly, the Constraint instances and bus interaction messages seem off:
  - Metrics viewer: 6.14B constraint instances, 1.44B bus interaction messages
  - APC analyzer: 896M constraint instances, 797M bus interaction messages

I wouldn't expect these numbers to be exactly the same: The APC analyzer takes each executed instruction and multiplies it with its cost (e.g. number of columns, constraints, or bus interactions). The metrics viewer computes what was actually in the proof. So:
- Any non-instruction AIRs are not included in the APC analyzer. But according to the metrics viewer (Trace cells by AIR pie chart), the top 7 AIRs by trace area are instruction AIRs and the "other" category accounts for only 3.3% of trace area.
- The APC analyzer does not expect the account for segmentation (a.k.a. sharding, splitting the computation into several chunks). But in my understanding, sharding should not increase the trace area in instruction AIRs, only in AIRs like the Poseidon chip, used for Merkelizing memory.
- Potentially there is padding happening, rounding the number of rows to the next power of 2. But this would at most double the numbers.

It would be great to understand these descrepancies.
